HANDOVER NOTE / TOPIC-SCOPED AGENT PROFILES AND PERSISTENT TOPIC CONTEXT

КОНТЕКСТ

Этот документ фиксирует состояние разработки фичи для Telegram forum topics, где каждый topic может работать как отдельный агент с собственной ролью, ограничениями по инструментам и persistent context.

Цель фичи:
1. Дать topic отдельную роль через agent profile.
2. Дать topic отдельный persistent context по аналогии с AGENT.md, но scoped на topic.
3. Подготовить архитектуру к Stage 3: SSH MCP tools, topic infra config, secret refs и approval workflow для опасных операций.

Состояние на момент передачи:
- Stage 1 завершен: profile-aware execution и tool policy hooks реализованы.
- Stage 2 завершен: persistent topic context реализован в storage, manager control plane и prompt assembly.
- Stage 3 еще не реализован.

БАЗОВАЯ МОДЕЛЬ

Разделение ответственности сейчас такое:
- AgentProfileRecord = переиспользуемая роль агента, prompt instructions, allow/block policy по tools.
- TopicContextRecord = отдельный persistent prompt/context для конкретного topic.
- TopicBindingRecord = связка topic -> agent_id.
- AgentExecutionProfile = runtime-объект, который собирается перед выполнением задачи и подается в executor.

ЧТО УЖЕ РЕАЛИЗОВАНО

STAGE 1
- Парсинг agent profile из JSON payload в runtime policy и prompt instructions.
- ToolAccessPolicyHook блокирует tools по allowlist/blocklist.
- LLM видит только tools, разрешенные текущему execution profile.
- Agent mode в Telegram перед выполнением задачи подгружает agent profile по route.agent_id.

STAGE 2
- Добавлен TopicContextRecord в persistent storage.
- Добавлены storage API для topic context: get, upsert, delete.
- Добавлены manager tools:
  topic_context_upsert
  topic_context_get
  topic_context_delete
  topic_context_rollback
- Topic context подмешивается в execution prompt как отдельная секция Persistent topic context.
- При cleanup forum topic теперь удаляется и связанный topic context.

PROMPT СБОРКА СЕЙЧАС

Execution prompt сейчас может состоять из трех секций:
- Profile instructions
- Topic instructions
- Persistent topic context

Topic instructions приходят из topic route override.
Persistent topic context приходит из TopicContextRecord.
Profile instructions приходят из AgentProfileRecord.profile.systemPrompt или system_prompt.

TOOL POLICY СЕЙЧАС

Поддерживаются такие ключи в JSON профиле:
- systemPrompt
- system_prompt
- allowedTools
- allowed_tools
- blockedTools
- blocked_tools

Поведение:
- если есть allowedTools, все остальные tools скрываются от модели и блокируются hook-ом;
- blockedTools всегда имеют приоритет и блокируются даже если есть allowlist.

ЧТО ХОТИМ СДЕЛАТЬ В STAGE 3

Главная цель Stage 3: дать selected topic-агентам доступ к SSH MCP tools для управления инфраструктурой, но без смешивания secrets с памятью агента или prompt context.

Целевой scope Stage 3:
1. Добавить TopicInfraConfigRecord.
2. Хранить secret refs отдельно от TopicContextRecord.
3. Подключить SSH MCP provider в agent tool ecosystem.
4. Добавить approval workflow для опасных SSH-команд.

ПРЕДЛАГАЕМАЯ МОДЕЛЬ ДАННЫХ ДЛЯ STAGE 3

TopicInfraConfigRecord:
- user_id
- topic_id
- target_name
- host
- port
- user
- auth_mode
- secret_ref
- sudo_secret_ref
- environment
- tags
- allowed_tool_modes
- approval_policy_id или inline approval policy

Секреты хранить НЕ в agent memory и НЕ в prompt context.
Секреты хранить отдельно, например в S3 под отдельным prefix, только через refs.

Пример логического разделения:
- TopicContextRecord = инструкции и операционные правила
- TopicInfraConfigRecord = куда подключаться и какие refs использовать
- Secret storage = private key, password, sudo password, tokens

SSH MCP ИНТЕГРАЦИЯ, КОТОРУЮ СТОИТ ДЕЛАТЬ

Рекомендуемый путь:
- отдельный provider, например crates/oxide-agent-core/src/agent/providers/ssh_mcp.rs
- запуск ssh-mcp как local stdio subprocess
- один process/session manager на topic или на active agent session
- передавать в ssh-mcp env для password-based creds
- для private key на MVP можно создавать временный файл 0600 из secret storage, потом удалять

Инструменты, которые можно отложить на второй проход Stage 3:

APPROVAL WORKFLOW ДЛЯ STAGE 3

Желательное поведение:
- безопасные команды идут без подтверждения;
- опасные команды требуют approve через Telegram inline action;
- approve должен быть topic-scoped и short-lived;
- после approve provider повторяет тот же tool call с approval token.

Что считать опасным минимум:
- sudo-exec
- rm -rf
- systemctl stop/restart
- reboot / shutdown
- docker compose down
- kubectl delete
- terraform apply / destroy
- file edit внутри /etc, ~/.ssh, systemd, nginx, db configs

Технически лучше делать approve не только hook-ом, а как runtime flow:
- policy/hook решает что нужен approve
- provider возвращает approval_required payload
- transport рендерит approve/reject UI
- callback сохраняет approval token
- повторный вызов инструмента проходит с токеном

РИСКИ И ОГРАНИЧЕНИЯ ДЛЯ STAGE 3

- Нельзя класть секреты в AgentMemory, TopicContextRecord, AgentProfileRecord или audit payload.
- Нужно санитизировать audit и logs, чтобы не утекали secret values.
- Background jobs у ssh-mcp держатся в памяти процесса, поэтому надо продумать lifecycle process manager.
- Нужна изоляция по topic, чтобы один topic не видел чужой target и чужие credentials.
- Нужно явно маркировать environment, чтобы prod и stage не перепутались в prompt.

ЖЕЛАТЕЛЬНЫЙ ПОРЯДОК РАБОТ НА STAGE 3

1. Добавить TopicInfraConfigRecord и manager tools для него.
2. Добавить secret_ref модель и contract для secret storage.
3. Сделать ssh_mcp provider без approval, только с жестким tool gating.
4. Добавить approval policy и interactive approve flow.
5. Потом расширять UX и inventory.

ЧТО НЕ ЗАБЫТЬ ПРИ ПРОДОЛЖЕНИИ
