What changed in code
Telegram integration and runtime UX:
- Agent task submission path now provides clearer background-start feedback with TaskId and preserves completion/error visibility for async execution
- Agent Mode entrypoints are rollout-gated and degrade to Chat Mode when disabled
- revoked users are force-downgraded from stale Agent Mode paths (persisted restore, /start blocked-by-task, and direct agent message handling)

Delegation safety:
- delegation depth is tracked explicitly in agent context and carried across sub-agent execution boundaries
- delegation beyond depth 2 is blocked before sub-agent creation with diagnostic error/logging

LLM concurrency and provider API:
- global and class-aware limiter controls introduced for LLM calls with user-facing reservation and background throttling
- background-class call paths aligned for narrator/loop-detection/tool flows, with telemetry on permit wait/saturation
- LlmProvider trait migrated to request-struct method signatures (ChatCompletionRequest and ChatWithToolsRequest)
- old too-many-arguments clippy suppression removed from provider boundary

Rollout and docs:
- AGENT_MODE_ENABLED added to configuration and operator surfaces
- Stage 6 rollout runbook added: docs/stage-6-slice-6-4-rollout.md
- README, README-ru and AGENTS.md updated with Stage 6 rollout context and landmarks

Operational configuration
Agent Mode v2 staged rollout is now controlled by:

Environment variables (.env):
- AGENT_MODE_ENABLED=true|false (default: false)
- WEB_OBSERVER_ENABLED=true|false (default: false, optional read-only web monitoring)

Docker Compose:
- AGENT_MODE_ENABLED passed through from .env
- WEB_OBSERVER_* remains optional and independent of core task recovery/runtime paths

Rollback notes:
- setting AGENT_MODE_ENABLED=false blocks new Agent Mode activation and degrades UI to Chat Mode
- runtime/task recovery is intentionally left operational for existing tasks
- rollback does not require manual persisted task/state repair
